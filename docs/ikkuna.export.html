class ikkuna.export.export.Exporter(frequency=1)¶

Bases: object

Class for managing publishing of data from model code.

An Exporter is used in the model code by either explicitly registering modules for tracking with Exporter.add_modules() or by calling it with newly constructed modules which will then be returned as-is, but be registered in the process.

e = Exporter(...)
features = nn.Sequential([
    nn.Linear(...),
    e(nn.Conv2d(...)),
    nn.ReLU()
])

No further changes to the model code are necessary, but for certain visualizations, the exporter requires access to the model in its entirety, so Exporter.add_model() should be used.

list – All tracked modules

_layer_counter¶

_layer_names¶

_weight_cache¶

class ikkuna.export.magic.WatchedParameter(baseObject, name=None)¶

Bases: torch.nn.parameter.Parameter

Wrapper class in order to trap mutations to a tensor inside a parameter. This is necesssary to retrieve updates to weights as applied by the optimizer, not just the raw gradients.

We cannot simply set the param.data field directly since whenever it is accessed, torch creates a new python object referencing the data, which is then returned, meaning our wrapper will not actually get notified when stuff happens since the updates are being done on a new object with a reference to the binary data. Thus we need to start one level above, at the Parameter.

_name¶

str – Name for the parameter, usually the layer identifier

ikkuna.export.magic.patch_mutator(method)¶

Patch mutating torch.Tensor methods used by builtin optimizers so that the changes made to the tensor can be recorded. Permissible methods must be bound (retrieved via Tensor object, not class) and are add_, addcdiv_, and mul_ at this time. The torch.nn.Parameter or torch.Tensor object to which the method is bound must posses a _label attribute (must be patched externally) in order to identify the layer which the weights belong to.

Note

I tried subclassing torch.nn.Parameter and torch.Tensor in order to simply trap calls to the mutating methods, and replace the values in the torch.nn.Module s containing them, but this fails either because a new torch.Tensor object is created on each access or because there’s weird error hinting at device mismatch between my wrapper class and the update tensors. Forwarding each and every property access to the wrapped tensor was not sufficient.

Parameters:method (MethodType) – Bound method to wrap
Raises:ValueError – If method is unbound or bound to object without a _label attribute.

class ikkuna.export.subscriber.SynchronizedSubscription(subscriber, kinds, tag=None)¶

Bases: ikkuna.export.subscriber.Subscription

A subscription which buffers messages and publishes a set of messages, each of a different kind, when one round (a train step) is over. This is useful for receiving several kinds of messages in each train step and always have them be processed together.
